Topic 01

DDIC — Data Dictionary

SE11: Central metadata repository for all data definitions — tables, structures, views, domains, search helps, and more.

Overview of DDIC

ABAP Dictionary (DDIC) is a central data management system in SAP. Its main function is to support the creation and management of data definitions — Data Elements, Structures, Table Types, etc.

📌 Transaction Code: SE11
Path: SAP Easy Access → Tools → ABAP Workbench → Development → ABAP Dictionary

Components of DDIC

  • (a) Database Tables
  • (b) Structures
  • (c) Views
  • (d) Data Elements
  • (e) Domains
  • (f) Search Help
  • (g) Lock Objects
  • (h) Table Maintenance Generator (TMG)

A. Database Tables

A table stores data in rows (Records) and columns (Fields). Custom table names must start with 'Z' or 'Y'.

Primary Key

A Primary Key uniquely identifies each row in a table by using one or more columns. No two rows can have the same key value.

Foreign Key

A Foreign Key establishes a relationship between different tables in the ABAP Dictionary. It maintains referential integrity between tables.

B. Structures

A Structure is a data object made up of components stored one after another in memory. It is a skeletal view of a table — contains column definitions but has no data. Also called a 'Work Area'. A structure may hold only a single record at run-time.

C. Views

A View is similar to a database table but does not contain physical data. It is a snapshot created by combining fields from single or multiple tables (called base tables).

D. Data Elements

A Data Element provides semantic (business) information of a database table field — it describes how a field is displayed to the end-user. It stores field labels, short text, and documentation.

📌 Relationship: Field → Data Element → Domain

E. Domain

A Domain describes the technical characteristics of an attribute — data type and length. It is assigned to a Data Element and can be reused across multiple fields. It also describes the value range of a field (Fixed Values or Value Table).

F. Search Help

A Search Help is a tool to help users find data in SAP. Accessed via the F4 key.

TypeDescription
Elementary Search HelpDescribes a single search path. Defines where data of the hit list is read from.
Collective Search HelpCombines several elementary search helps, offering multiple alternative search paths.

G. Lock Objects

Lock Objects prevent simultaneous access by multiple users. They synchronize access to the same data. A lock object auto-generates two function modules:

ENQUEUE_<LockObjectName> " To insert in queue (LOCK) DEQUEUE_<LockObjectName> " To remove from queue (UNLOCK)
Lock TypeCodeBehavior
Read Lock (Shared)SAllows multiple users to read simultaneously
Write Lock (Exclusive)EPrevents any other user from accessing
Enhanced Write LockXStricter exclusive lock; cannot be accumulated

H. Table Maintenance Generator (TMG)

TMG creates a maintenance interface for database tables without writing ABAP code. Can be checked via SM30 or SM31.

TypeDescription
Single StepOne screen where user can add, edit, delete records directly
Two StepFirst screen shows overview (key fields); clicking opens a detail screen for full editing

Buffering

When an Open SQL statement reads a record, if the table is buffered, the record is saved in RAM on the application server. Later reads fetch from buffer instead of the database — improving performance.

TypeBehavior
Single Record BufferedFetch and cache one record at a time
Generic Area BufferedBuffer records based on a partial key (WHERE condition prefix)
Fully BufferedEntire table is loaded into the buffer at once

💡 Buffering is best for master data tables — updated rarely, accessed frequently.

Interview Q&As

QWhat is DDIC in SAP? What is its transaction code?
ADDIC (Data Dictionary) is the central metadata repository in SAP. It manages all data definitions like tables, structures, views, data elements, and domains. It is the single source of truth for all data definitions used across the SAP system. Transaction code: SE11.
QWhat is the difference between a Transparent, Pooled, and Cluster Table?
ATransparent: One-to-one mapping with DB table, same name/fields, stores master data, supports secondary indexes. Pooled: Many-to-one, stored in Table Pool, different names/fields, stores small internal control data, no secondary index. Cluster: Many-to-one, stored in Table Cluster, used when tables share primary keys and are accessed together, no secondary index.
QWhat is a Data Element? How is it different from a Domain?
AData Element provides semantic (business) meaning to a field — it stores field labels, short text, and documentation. Domain defines the technical characteristics — data type, length, and value range. A Data Element uses a Domain for its technical definition. Multiple data elements can share the same domain.
QExplain the relationship between Field, Data Element, and Domain.
AField → Data Element → Domain. A table field references a Data Element which provides its business description and labels. The Data Element references a Domain which provides the technical type, length, and valid value ranges. This three-layer architecture promotes reusability and consistency.
QWhat are the 4 types of Views in SAP DDIC?
A(1) Database View: Inner join on multiple tables — read only. (2) Projection View: Single table, read/write, shows only selected fields. (3) Help View: Outer join, multiple tables, used as search help selection method — read only. (4) Maintenance View: Multiple tables with FK relationship, full read/write/delete capability.
QWhat is a Lock Object? What are the types of locks?
AA Lock Object controls concurrent access to database records. It generates two FMs: ENQUEUE (to lock) and DEQUEUE (to unlock). Types: S — Shared (Read) Lock: allows multiple simultaneous reads. E — Exclusive (Write) Lock: prevents all other access. X — Enhanced Write Lock: exclusive lock that cannot be accumulated (cumulated).
QWhat is buffering in SAP? What are its types?
ABuffering stores frequently-accessed table data in application server memory (RAM) to avoid repeated database reads. Types: (1) Single Record Buffering — one record cached at a time. (2) Generic Area Buffering — records matching a partial key prefix are buffered together. (3) Full Buffering — entire table loaded into buffer. Best for small, rarely-changed master data tables.
QWhat is the difference between Primary Index and Secondary Index?
APrimary Index: Created automatically based on primary key fields. Ensures uniqueness and fast access by primary key. Secondary Index: Manually created on non-key fields to speed up queries using those fields in WHERE conditions. Up to 16 secondary indexes per table. Too many secondary indexes cause overhead on INSERT/UPDATE operations.
